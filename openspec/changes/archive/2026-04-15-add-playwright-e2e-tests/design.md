## Context

ダッシュボードトップ（`/`）は Next.js の Server Component でレンダリングされ、Prisma 経由でDBから記事一覧を取得する。クライアントサイドのインタラクション（スクレイピング・フィルタリング・既読切替）はそれぞれ以下のAPIを呼び出す。

| 操作 | API | 方式 |
|---|---|---|
| 記事取得ボタン | `POST /api/scrape` | client fetch |
| 記事フィルタ | URL クエリパラメータ変更 → SSR再レンダリング | router.push |
| 既読切替 | `PATCH /api/articles/[id]/read` | client fetch（楽観的更新） |
| お気に入り切替 | `PATCH /api/articles/[id]/favorite` | client fetch（楽観的更新） |

SSR で初期データを取得するため、初期表示にはDBへのアクセスが必要。

## Goals / Non-Goals

**Goals:**
- 「記事を取得」ボタンのローディング状態・完了後の結果表示を検証する
- フィルターボタンクリックで URL クエリパラメータが正しく更新されることを検証する
- 記事タイトルクリックで既読状態に変わり、カードのスタイルが更新されることを検証する
- テストは CI でも安定実行できるよう、DBへの副作用をなるべく排除する

**Non-Goals:**
- 全フィルター組み合わせの網羅テスト
- DB の実データを使った統合テスト（本変更のスコープ外）
- モバイルレイアウトのテスト

## Decisions

### 1. クライアントサイドAPIはPlaywright `page.route()` でモックする

`POST /api/scrape` と `PATCH /api/articles/[id]/read` は `page.route()` でインターセプトし、固定レスポンスを返す。

**理由**: テストDBのセットアップなしにAPIの振る舞いを制御でき、CI でのシード管理が不要になる。DBへの副作用もなく冪等にテストを実行できる。

**代替案**: テストDB + シードスクリプト → セットアップコストが高く、並行テスト時に競合リスクがある。

### 2. SSR 初期データは `page.route()` でページ HTML ではなくAPIレスポンスを差し替える方針を採らず、空状態でテストする

フィルターのURLパラメータ変更テストは、記事が0件の空状態ページでも検証できる（URLパラメータの更新自体を検証するため）。記事データが必要な既読切替テストのみ、`page.route()` で `GET /` へのレスポンスを差し替えるのではなく、`page.route()` で PATCH エンドポイントをモックし、Playwright の `page.evaluate()` で DOM に直接初期状態を注入するか、あるいはテスト用シードを使う。

実際の方針として、**既読切替テストのみテスト用DBシードに依存し、それ以外はモックのみで完結させる**。

**理由**: SSR ページのHTMLレスポンスを丸ごとモックするのは保守コストが高い。スクレイプボタンとフィルターはDBデータ不要で検証可能。

### 3. テストファイルは `e2e/dashboard.spec.ts` の1ファイルに集約する

**理由**: テスト対象が単一画面（`/`）のため、ファイル分割のメリットが少ない。将来的に画面が増えたタイミングでディレクトリ構成を見直す。

### 4. `data-testid` 属性でセレクタを安定化する

既存コンポーネントに `data-testid` を追加し、クラス名や表示テキストの変更に対してテストを堅牢にする。

## Risks / Trade-offs

- **[リスク] SSRデータのモック困難** → フィルター結果の表示内容検証は E2E では行わず、URLパラメータ変更とフィルターボタンのアクティブ状態変化の検証に留める
- **[リスク] 楽観的更新の競合** → 既読切替テストでは PATCH モックのレスポンスタイミングを制御し、楽観的更新後の状態を検証する
- **[リスク] `data-testid` 追加の保守コスト** → テストIDはコンポーネント変更時に一緒にレビューする運用とする
