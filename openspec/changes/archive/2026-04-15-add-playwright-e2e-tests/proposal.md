## Why

ダッシュボードの主要インタラクション（記事取得・フィルタリング・既読マーク）に自動化されたE2Eテストが存在しないため、UIレグレッションを手動でしか検知できない。Playwright を導入してトップページの重要シナリオをカバーし、変更時の品質担保を自動化する。

## What Changes

- `e2e/` ディレクトリを新設し、Playwright テストファイルを追加する
- テスト対象シナリオ：
  - 「記事を取得」ボタンクリック → `/api/scrape` へ POST リクエストが送信され、ローディング表示後に結果が表示される
  - ソースフィルター（すべて / Zenn / Qiita / GitHub / HN）クリック → URL クエリパラメータ `source` が更新され、表示記事が絞り込まれる
  - 状態フィルター（すべて / 未読 / 既読 / お気に入り）クリック → URL クエリパラメータ `status` が更新される
  - 記事タイトルリンクのクリック → 未読記事が既読状態（`isRead: true`）に変わり、カードのスタイルが変化する

## Capabilities

### New Capabilities

- `dashboard-scrape-button`: 「記事を取得」ボタンのE2Eテスト — クリックでスクレイピングAPIが呼ばれ、ローディング・結果表示が正しく動作することを検証する
- `dashboard-filter`: フィルターバーのE2Eテスト — ソース・状態フィルターのクリックでURLパラメータが更新され、記事一覧が正しく絞り込まれることを検証する
- `dashboard-read-toggle`: 記事タイトルクリックによる既読状態変更のE2Eテスト — クリック後にカードのスタイルと状態ボタンが「既読」表示に切り替わることを検証する

### Modified Capabilities

（なし）

## Impact

- **新規ファイル**: `e2e/dashboard.spec.ts`
- **設定**: `playwright.config.ts`（既存）、`package.json` の `test:e2e` スクリプト（既存）
- **依存**: `@playwright/test`（インストール済み）、Chromium ブラウザ（インストール済み）
- **APIモック**: `/api/scrape` はテスト時にネットワークインターセプトでモックし、DBへの副作用なしにテストを実行する
- **DBモック**: 記事一覧・既読状態変更の `/api/articles` エンドポイントもインターセプトしてモックデータを返す
